## Kubernetes는 클라우드 운영체제다

이 장에서 가장 먼저 인상 깊었던 표현은 Kubernetes를 **클라우드 운영체제**처럼 바라보는 관점이었다.

Linux가 하드웨어 위에서 프로세스, 파일 시스템, 네트워크를 추상화하듯이, Kubernetes는 인프라 위에서 다음과 같은 요소들을 추상화한다.

- 배포
- 네트워킹
- 스토리지
- 보안
- 서비스 운영

이 관점에서 Kubernetes는 단순히 컨테이너를 실행하는 도구가 아니다.  
여러 인프라 환경 위에서 애플리케이션을 일관된 방식으로 배포하고 운영하기 위한 공통 계층에 가깝다.

따라서 Kubernetes를 이해하면 특정 클라우드 서비스 하나를 익히는 것을 넘어, 다양한 클라우드 인프라를 더 일관된 방식으로 이해할 수 있다.

---

## GitOps란 무엇인가

GitOps는 Git 저장소를 **Single Source of Truth**, 즉 단일 진실 공급원으로 삼아 원하는 인프라 상태를 선언적으로 관리하는 방법론이다.

핵심은 간단하다.

> 원하는 상태를 Git에 선언하면, 도구가 실제 인프라를 그 상태에 맞춘다.

기존에는 사람이 직접 배포 명령을 실행하거나 클러스터에 접속해 설정을 변경하는 경우가 많았다.  
반면 GitOps에서는 Git에 기록된 선언이 기준이 된다.

즉, Git에 있는 내용이 “원하는 상태”이고, 실제 클러스터는 그 상태를 따라가야 한다.

---

## GitOps의 기본 흐름

GitOps의 흐름은 다음과 같이 정리할 수 있다.

```text
개발자가 Manifest 작성
→ Git 저장소에 Push
→ GitOps 도구가 변경 감지
→ 클러스터에 자동 반영
→ 실제 상태와 Git 상태의 차이를 지속적으로 감지
```

조금 더 구체적으로 보면 다음과 같다.

1. 개발자가 `Deployment`, `Service` 같은 Kubernetes Manifest를 작성한다.
2. 작성한 파일을 Git 저장소에 Push한다.
3. Argo CD나 Flux 같은 GitOps 도구가 Git 저장소를 감시한다.
4. 새로운 변경이 감지되면 클러스터에 자동으로 적용한다.
5. 누군가 클러스터에서 직접 설정을 변경하면, Git에 선언된 상태와 다르다는 것을 감지한다.
6. GitOps 도구는 실제 클러스터 상태를 다시 Git에 선언된 상태로 맞춘다.

여기서 중요한 점은 **Git이 기준 상태**라는 것이다.

클러스터에서 직접 변경된 내용은 임시 변경일 뿐이고, Git에 선언되지 않았다면 원하는 상태가 아니다.  
이 구조 덕분에 인프라 변경을 더 추적 가능하고 재현 가능한 방식으로 관리할 수 있다.

---

## GitOps의 장점

GitOps의 장점은 단순히 배포 자동화에만 있지 않다.

### 1. 배포 자동화

Git에 Push하면 배포가 진행된다.  
별도의 배포 명령을 사람이 직접 실행하지 않아도 된다.

### 2. 배포 히스토리 관리

Git 히스토리가 곧 배포 히스토리가 된다.

누가, 언제, 무엇을 변경했는지 Git 로그를 통해 확인할 수 있다.

### 3. 롤백

잘못된 배포가 발생했을 때 Git 이력을 이용해 이전 상태로 되돌릴 수 있다.

예를 들어 `git revert`를 사용하면 특정 변경을 되돌릴 수 있고, GitOps 도구는 되돌려진 선언 상태를 기준으로 클러스터를 다시 동기화할 수 있다.

### 4. 코드 리뷰

인프라 변경도 Pull Request를 통해 리뷰할 수 있다.

애플리케이션 코드처럼 인프라 변경도 동료 검토를 거친 뒤 반영할 수 있다는 점이 중요하다.

### 5. 재현 가능성

Git에 선언된 내용을 기준으로 새로운 환경을 만들 수 있다.

개발, 스테이징, 운영 환경을 일관성 있게 구성하거나, 장애 상황에서 환경을 재구성할 때 도움이 된다.

## 기존 CI/CD 방식과 GitOps를 비교해보고 싶은 점

이 장을 읽으면서 현재 일반적으로 많이 사용하는 CI/CD 기반 배포 방식과 GitOps 방식의 차이를 비교해보고 싶어졌다.

예를 들어 많은 환경에서는 다음과 같은 방식으로 배포가 이루어진다.

```text
Git Push
→ CI/CD Pipeline 실행
→ helm upgrade 또는 kubectl apply 실행
→ 클러스터에 배포
```

이 방식도 Git을 활용한다는 점에서는 GitOps와 닮아 있다.  
하지만 GitOps 도구를 사용하는 방식과는 차이가 있다.

| 비교 항목 | 일반적인 CI/CD 기반 배포 | GitOps 기반 배포 |
|---|---|---|
| 배포 트리거 | Pipeline 실행 | Git 변경 감지 |
| 배포 방식 | 스크립트나 명령 실행 중심 | 선언된 상태와 실제 상태 동기화 |
| 상태 관리 | 배포 시점 중심 | 지속적인 상태 감시 |
| 기준 상태 | Pipeline 결과 또는 배포 스크립트 | Git 저장소 |
| Drift 감지 | 별도 구현 필요 | Git 상태와 클러스터 상태 차이 감지 |
| 롤백 | Pipeline 또는 수동 절차 | Git 이력 기반 |
| 리뷰 절차 | 구성에 따라 다름 | Pull Request 흐름과 자연스럽게 연결 가능 |

특히 관심이 가는 부분은 Pull Request 기반 승인 절차를 인프라 변경 흐름에 어떻게 연결할 수 있는지이다.

AI가 인프라 변경을 도와주는 환경에서는 자동화 자체보다 다음 질문이 더 중요해진다.

- 변경 사항이 Git에 기록되는가?
- 사람이 변경 내용을 검토할 수 있는가?
- 승인되지 않은 변경이 실제 환경에 반영되지 않도록 막을 수 있는가?
- 문제가 생겼을 때 이전 상태로 되돌릴 수 있는가?
- 변경 이력을 감사 가능하게 남길 수 있는가?

GitOps는 이런 질문에 답하기 좋은 구조를 제공할 수 있다.

## 문서화는 왜 중요한가

이 장에서 공감이 많이 갔던 부분은 문서화에 대한 이야기였다.

모든 과정을 문서화하는 것은 중요하지만, 현실에서는 가장 먼저 빠지는 일이 문서화다.

“나중에 정리해야지”라고 미뤄두면 시간이 지나면서 왜 그렇게 설정했는지 본인도 기억하지 못한다.  
새로운 팀원이 들어왔을 때 “이 설정은 왜 이렇게 되어 있나요?”라는 질문에 답하지 못하는 상황도 생긴다.

결국 문서화가 부족하면 사람이 바뀔 때마다 같은 시행착오를 반복하게 된다.

문서화는 단순히 결과를 기록하는 일이 아니다.  
특히 인프라와 운영 영역에서는 **의사결정의 이유**를 남기는 것이 중요하다.

예를 들어 다음과 같은 내용이 문서화되어야 한다.

- 왜 이 도구를 선택했는가
- 어떤 대안이 있었는가
- 어떤 기준으로 결정했는가
- 어떤 설정을 변경했는가
- 변경 전후에 무엇을 검증해야 하는가
- 장애가 발생하면 어떻게 대응해야 하는가
- 나중에 되돌리려면 어떤 절차가 필요한가

AI 도구를 활용하면 문서화의 진입 장벽은 낮아질 수 있다.  
하지만 중요한 것은 문서화를 별도의 일이 아니라 작업 흐름 안에 자연스럽게 포함시키는 것이다.

## GitAIOps란 무엇인가

GitAIOps는 GitOps의 자동화된 상태 관리에 AI의 실행 능력을 더한 개념이다.

GitOps가 다음과 같은 방식이라면,

```text
Git에 선언하면 자동으로 반영된다.
```

GitAIOps는 한 단계 더 나아가 다음과 같은 방식에 가깝다.

```text
자연어로 지시하면 Git에 선언되고 자동으로 반영된다.
```

즉, 사람이 직접 YAML, Helm Chart, Kubernetes Manifest를 작성하는 대신 자연어로 의도를 설명하면, AI가 이를 선언 파일로 변환하거나 기존 파일을 수정한다.  
그리고 그 변경 사항이 Git에 기록되고, GitOps 도구가 실제 인프라에 반영한다.

흐름을 정리하면 다음과 같다.

```text
자연어 지시
→ AI 에이전트가 의도 해석
→ Manifest / Helm Chart / 설정 파일 생성 또는 수정
→ Git에 변경 사항 기록
→ 리뷰와 승인
→ GitOps 도구가 실제 인프라에 반영
```

여기서 중요한 점은 AI가 곧바로 운영 환경을 수정하는 것이 아니라, Git이라는 통제 가능한 흐름 안에 변경 사항을 남긴다는 것이다.

AI가 인프라 작업을 도와줄수록 Git은 더 중요해질 수 있다.  
Git은 변경 이력, 리뷰, 승인, 롤백, 감사의 기준점이 될 수 있기 때문이다.

## Notiflex 시나리오를 보며 든 생각

책에서는 `Notiflex`라는 B2B 알림 SaaS 플랫폼 시나리오가 등장한다.

Notiflex는 고객사의 서비스에서 발생하는 이벤트를 받아 이메일, SMS, 푸시 알림으로 발송하는 서비스다.

예를 들면 고객사 서비스에서 다음과 같은 이벤트가 발생할 수 있다.

- 회원가입
- 결제 완료
- 배송 시작
- 배송 완료
- 비밀번호 변경
- 중요 공지 발송

고객사는 Notiflex의 API를 호출하기만 하면 되고, 알림 발송에 필요한 복잡한 처리는 Notiflex가 담당한다.

이 시나리오를 보면서 공통 알림 플랫폼의 필요성을 생각해볼 수 있었다.

여러 서비스가 각각 이메일, 문자, 푸시 알림 기능을 따로 구현하면 중복이 많아지고 관리가 어려워진다.  
반대로 공통 알림 플랫폼을 두면 각 서비스는 이벤트만 전달하고, 알림 발송 정책과 채널 처리는 한 곳에서 관리할 수 있다.

이는 알림뿐만 아니라 승인, 결재, 이벤트 처리 같은 공통 업무 프로세스에도 확장해볼 수 있는 구조라고 느꼈다.

## 기술은 목적이 아니라 문제 해결을 위한 수단이다

이 장에서 특히 공감했던 메시지는 기술 도입의 이유에 대한 부분이었다.

중요한 것은 기술적 이유 자체가 아니라, 비즈니스가 성장하면서 **그 목적에 맞는 도구들이 필요해졌다**는 점이다.

기술은 문제를 해결하기 위해 도입하는 것이지, 기술 자체가 목적이 되어서는 안 된다.

AI를 활용한 개발 흐름에서도 이 점은 중요하다.

AI가 많은 것을 도와줄 수 있게 되면 처음부터 완벽한 구조를 만들고 싶어진다.  
하지만 실제로 프로젝트를 진행해보면 처음부터 모든 것을 완벽하게 설계하는 것보다, 현재 해결해야 하는 문제에 맞춰 필요한 도구를 적절한 시점에 추가하는 것이 더 중요하다.

좋은 기술 선택은 최신 기술을 많이 쓰는 것이 아니다.

현재 문제를 가장 단순하고 안정적으로 해결할 수 있는 도구를 선택하는 것이다.

## Claude Code와 CLAUDE.md

Claude Code는 작업 디렉터리에 있는 `CLAUDE.md` 파일을 자동으로 읽고, 이를 동작 규칙으로 삼는다.

`CLAUDE.md`는 AI 에이전트에게 프로젝트의 규칙과 맥락을 알려주는 문서라고 볼 수 있다.

예를 들어 다음과 같은 내용을 정의할 수 있다.

- 어떤 파일을 참고해야 하는가
- 어떤 기준으로 도구를 선택해야 하는가
- 어떤 작업 절차를 따라야 하는가
- 어떤 명령은 조심해야 하는가
- 작업 결과를 어떻게 검증해야 하는가

책에서는 관련 문서를 다음과 같이 나누어 설명한다.

### decision-guides

의사결정 가이드이다.

주로 다음과 같은 질문에 답한다.

- 왜 이 도구를 선택했는가?
- 다른 대안은 무엇인가?
- 어떤 기준으로 비교했는가?
- 현재 상황에서 가장 적절한 선택은 무엇인가?

### prompt-guardrails

실행 가드레일이다.

AI가 작업을 수행할 때 지켜야 할 사전 조건과 절차를 정의한다.

예를 들면 다음과 같은 내용이 포함될 수 있다.

- 실행 전 확인해야 할 조건
- 단계별 실행 절차
- 금지해야 할 작업
- 장애 발생 시 트러블슈팅 방법
- 운영 환경에서 반드시 확인해야 할 사항

### result-templates

결과 템플릿이다.

AI가 작업을 완료한 뒤, 결과가 기대와 일치하는지 검증하기 위한 체크리스트 역할을 한다.

예를 들면 다음과 같은 내용을 포함할 수 있다.

- 배포가 정상적으로 되었는가?
- 필요한 리소스가 생성되었는가?
- 로그에 오류가 없는가?
- 모니터링 지표가 정상인가?
- 롤백이 필요한 상황은 아닌가?

이 구조는 단순히 AI에게 “작업을 해달라”고 요청하는 수준을 넘어선다.  
AI가 조직이나 프로젝트의 기준 안에서 일할 수 있도록 규칙과 검증 체계를 제공하는 방식이다.

---

## AI와 함께 일하는 흐름: 탐색, 비교, 실행

책에서 제안하는 AI와의 작업 흐름도 인상 깊었다.

```text
탐색 → 비교 → 실행
```

### 1. 탐색

먼저 가능한 선택지를 넓게 살펴본다.

예를 들면 다음과 같이 질문할 수 있다.

```text
모니터링 도구는 어떤 것들이 있을까?
```

이 단계에서는 바로 실행하지 않고, 가능한 대안을 확인하는 것이 중요하다.

### 2. 비교

그다음 선택지들을 비교한다.

```text
다른 대안은 없어? 각각 비교해줘.
```

이 단계에서는 장단점, 운영 비용, 학습 비용, 현재 환경과의 적합성 등을 비교할 수 있다.

### 3. 실행

마지막으로 선택한 방향으로 실행한다.

```text
그 방식으로 진행해줘.
```

이 흐름은 AI를 사용할 때 중요한 기준이 될 수 있다.

AI에게 처음부터 실행을 맡기기보다, 먼저 탐색하고 비교한 뒤 실행하게 해야 더 안전하고 납득 가능한 결과를 얻을 수 있다.

## 내가 이해한 방식

이 장을 읽고 나서 GitOps와 GitAIOps를 다음과 같이 이해했다.

```text
GitOps:
사람이 원하는 인프라 상태를 Git에 선언한다.
도구가 실제 인프라를 Git 상태에 맞춘다.

GitAIOps:
사람이 자연어로 원하는 상태를 설명한다.
AI가 그 의도를 Git에 선언 가능한 형태로 만든다.
도구가 실제 인프라를 Git 상태에 맞춘다.
```

즉, GitAIOps는 GitOps를 대체하는 개념이라기보다 GitOps 위에 AI 에이전트를 더하는 방식에 가깝다.

기존에는 사람이 직접 Manifest나 Helm Chart를 작성했다면, 앞으로는 AI가 그 작업을 도와줄 수 있다.  
다만 AI가 작성한 결과도 반드시 Git에 기록되고, 리뷰와 검증을 거쳐야 한다.

AI 시대의 인프라 운영에서 중요한 것은 AI가 얼마나 많은 일을 할 수 있는지가 아니다.

중요한 것은 AI가 만든 변경을 어떻게 기록하고, 검토하고, 승인하고, 되돌릴 수 있는지이다.

## 정리

Chapter 1을 읽으면서 가장 크게 남은 키워드는 다음과 같다.

- Kubernetes는 클라우드 운영체제처럼 볼 수 있다.
- GitOps는 Git을 인프라 상태의 단일 진실 공급원으로 삼는다.
- GitAIOps는 GitOps에 AI 에이전트와 자연어 인터페이스를 더한 흐름이다.
- AI가 인프라를 다룰수록 Git 기반의 기록, 검토, 승인, 롤백 구조가 중요해진다.
- 문서화는 결과만이 아니라 의사결정의 이유를 남기는 일이다.
- 기술은 목적이 아니라 문제 해결을 위한 수단이다.
- AI와 함께 일할 때는 탐색, 비교, 실행의 흐름이 유용하다.

이번 장은 단순히 새로운 도구를 소개하는 내용이라기보다, AI 시대에 개발자가 인프라를 어떤 관점으로 바라봐야 하는지 설명하는 장처럼 느껴졌다.

AI는 코드 작성뿐만 아니라 인프라 설계와 운영 과정에도 점점 더 깊이 들어오게 될 것이다.  
그럴수록 개발자는 인프라 자체를 이해하는 것뿐만 아니라, AI가 만든 변경을 안전하게 다루는 방법도 함께 익혀야 한다.

결국 앞으로 중요해질 역량은 다음과 같다고 느꼈다.

> 인프라를 직접 다루는 능력과  
> AI가 인프라를 안전하게 다루도록 규칙과 절차를 설계하는 능력
