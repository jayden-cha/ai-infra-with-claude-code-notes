## Claude Code 자동 승인 모드

Claude Code는 다음 명령으로 자동 승인 모드로 실행할 수 있다.

```bash
claude --dangerously-skip-permissions
```

이 옵션을 사용하면 파일 생성, 명령 실행, Git 작업 등을 매번 승인하지 않고 자동으로 진행할 수 있다.

편리하지만 이름 그대로 위험할 수 있는 옵션이다.  
AI가 명령을 실행하거나 파일을 수정할 때 매번 사용자의 확인을 받지 않기 때문이다.

따라서 자동 승인 모드는 실습 환경이나 안전한 개발 환경에서는 생산성을 높여줄 수 있지만, 운영 환경이나 민감한 리소스에 접근할 수 있는 환경에서는 신중하게 사용해야 한다.

### 생각한 점

AI 에이전트에게 작업을 맡길수록 중요한 것은 “얼마나 자동화할 것인가”가 아니라 **어디까지 자동화해도 되는가**이다.

자동 승인 모드를 사용한다면 최소한 다음 기준이 필요하다고 느꼈다.

- 자동 실행해도 되는 명령
- 반드시 사용자 승인이 필요한 명령
- 운영 환경에 영향을 줄 수 있는 명령의 제한
- Git 작업 자동화 범위
- 민감 정보 접근 제한
- 삭제, 배포, push 같은 되돌리기 어려운 작업에 대한 별도 확인 절차

AI 도구가 강력해질수록 권한의 경계를 명확히 정하는 것이 더 중요해진다.

## `/btw`를 활용한 빠른 질문

Claude Code에서는 `/btw`를 사용해 현재 작업 흐름을 방해하지 않고 빠른 질문을 던질 수 있다.

`/btw`로 입력한 질문은 대화 히스토리에 기록되지 않고, 답변은 오버레이로 표시된 뒤 사라진다.

이 기능은 현재 작업 컨텍스트를 유지하면서도, 중간에 떠오른 보조 질문을 빠르게 확인할 때 유용하다.

예를 들면 다음과 같은 상황에서 사용할 수 있다.

```text
/btw 이 명령의 옵션은 무슨 의미야?
/btw 지금 사용 중인 브랜치 확인 명령이 뭐였지?
/btw kubectl context 확인 명령 알려줘
```

### 생각한 점

AI와 긴 작업을 진행하다 보면 중간중간 작은 질문이 계속 생긴다.

그 질문을 모두 일반 대화로 남기면 컨텍스트가 길어지고, 본래 작업의 흐름이 흐려질 수 있다.  
`/btw`는 이런 보조 질문을 따로 처리할 수 있게 해주기 때문에 컨텍스트를 효율적으로 사용하는 데 도움이 될 것 같다.

## statusline으로 현재 상태 표시하기

Claude Code 하단에는 현재 작업 상태를 표시하는 `statusline`을 설정할 수 있다.

statusline을 잘 구성하면 작업 중 필요한 정보를 한눈에 확인할 수 있다.

예를 들어 다음과 같은 정보를 표시할 수 있다.

- 현재 작업 경로
- Git 브랜치
- 현재 모델명
- 컨텍스트 사용량
- API 사용률
- Kubernetes context
- namespace
- 개발/운영 환경 구분

특히 인프라 작업에서는 현재 어느 환경을 대상으로 작업하고 있는지 확인하는 것이 매우 중요하다.

잘못된 클러스터나 namespace에서 명령을 실행하면 큰 문제가 생길 수 있기 때문에, statusline은 단순한 편의 기능이 아니라 실수를 줄이는 안전장치로도 볼 수 있다.

### 생각한 점

AI 에이전트가 명령을 실행할 수 있는 환경에서는 현재 상태를 계속 눈에 보이게 만드는 것이 중요하다.

작업자는 “지금 어느 경로에서, 어떤 브랜치로, 어떤 context를 대상으로 작업하고 있는가”를 놓치지 않아야 한다.

statusline은 이런 정보를 지속적으로 보여주는 도구로 활용할 수 있을 것 같다.

## `!` 접두사로 셸 명령 직접 실행하기

Claude Code 프롬프트에서 명령 앞에 `!`를 붙이면 AI를 거치지 않고 셸 명령을 바로 실행할 수 있다.

예를 들면 다음과 같다.

```bash
!pwd
!git status
!kubectl get pods
```

이 방식은 AI에게 설명을 요청하지 않고, 현재 상태만 빠르게 확인하고 싶을 때 유용하다.

다만 실제 셸 명령이 직접 실행되므로, 영향이 큰 명령은 주의해서 사용해야 한다.

예를 들어 다음과 같은 명령은 바로 실행하기보다 먼저 영향 범위를 확인하는 것이 좋다.

```bash
!kubectl delete ...
!helm uninstall ...
!git push ...
!terraform destroy ...
```

## Claude Code 세션 이어가기

Claude Code는 이전 세션을 이어서 불러올 수 있다.

예를 들어 새 터미널을 열어 프로젝트 디렉터리로 이동한 뒤 다음과 같이 실행할 수 있다.

```bash
cd <PROJECT_DIR>
claude --continue --dangerously-skip-permissions
```

주요 옵션은 다음과 같다.

| 옵션 | 의미 |
|---|---|
| `--continue` 또는 `-c` | 현재 디렉터리의 가장 최근 대화를 바로 이어감 |
| `--resume` 또는 `-r` | 여러 세션 중 하나를 선택해서 이어감 |
| `--dangerously-skip-permissions` | 승인 절차를 건너뛰고 자동 진행 |

AI와 작업을 하다 보면 터미널을 닫았다가 다시 열거나, 다음 날 이어서 작업해야 하는 경우가 있다.  
이때 세션 이어가기를 사용하면 이전 맥락을 다시 설명하지 않아도 된다.

### 생각한 점

인프라 작업에서는 “어디까지 진행했는가”를 잃어버리는 것이 위험할 수 있다.

세션을 이어갈 수 있는 기능은 작업 흐름을 유지하는 데 도움이 된다.  
다만 세션에만 의존하기보다는, 중요한 진행 상황은 반드시 문서에도 남겨야 한다고 느꼈다.

## 되감기 기능과 체크포인트

Claude Code에는 되감기 기능이 있다.

Claude Code가 파일을 편집하기 직전마다 자동으로 체크포인트가 저장되기 때문에, 특정 프롬프트 이전 상태로 되돌릴 수 있다.

되감기 메뉴는 다음 방식으로 열 수 있다.

```text
Esc 키 두 번
또는
/rewind
```

되감기 시 되돌릴 대상을 선택할 수 있다.

| 선택 | 되돌리는 대상 |
|---|---|
| `Code only` | 파일 변경만 되돌리고 대화는 유지 |
| `Conversation only` | 대화만 해당 시점으로 되돌리고 파일은 유지 |
| `Code and conversation` | 파일과 대화를 모두 되돌림 |

이 기능은 AI가 원하지 않은 방향으로 파일을 수정했을 때 유용하다.

하지만 되감기가 모든 것을 되돌려주는 것은 아니다.

되감기로 되돌릴 수 있는 것은 주로 Claude Code가 편집한 작업 디렉터리 안의 파일이다.

반대로 다음과 같은 작업은 되돌릴 수 없다.

- 시스템 변경
  - 패키지 설치
  - 셸 설정 파일 수정
  - 홈 디렉터리나 시스템 영역 변경
- 클라우드 또는 클러스터 작업
  - 클러스터 생성
  - `kubectl apply`
  - 클라우드 리소스 생성/삭제
- 외부로 나간 작업
  - `git push`
  - 이미지 레지스트리 push
  - 외부 API 호출

즉, 로컬 파일 편집은 되감기로 복구할 수 있지만, 실제 인프라에 반영된 작업은 별도의 롤백 절차가 필요하다.

### 생각한 점

AI와 함께 작업할수록 “되돌릴 수 있는 작업”과 “되돌리기 어려운 작업”을 구분하는 것이 중요하다.

예를 들면 다음과 같이 나눠볼 수 있다.

| 작업 종류 | 되돌리는 방법 |
|---|---|
| 로컬 파일 변경 | Claude Code 되감기 |
| Git 커밋 | `git revert`, `git reset` |
| 원격 저장소 push | revert 커밋 또는 브랜치 전략 |
| Kubernetes 리소스 변경 | GitOps 롤백, `kubectl rollout undo`, 이전 manifest 적용 |
| 클라우드 리소스 변경 | IaC 롤백 또는 수동 복구 절차 |
| 이미지 레지스트리 push | 태그 정책, 이미지 보존 정책, 배포 차단 |

되감기 기능은 유용하지만, 운영에 영향을 준 변경까지 자동으로 되돌려주지는 않는다.  
따라서 실제 인프라 작업에서는 별도의 롤백 전략을 함께 준비해야 한다.

## `kubectl`과 Cloud SDK

책에서는 `kubectl`을 Cloud SDK를 통해 설치하고 관리하는 흐름을 다룬다.

이 방식의 장점은 크게 두 가지다.

1. `gcloud components update`를 통해 관련 도구 버전을 함께 관리할 수 있다.
2. GKE 인증에 필요한 플러그인을 함께 관리할 수 있다.

기본 프로젝트, 리전, 존을 설정해두면 이후 명령에서 `--project`나 `--zone` 같은 옵션을 매번 지정하지 않아도 된다.

예시는 다음과 같다.

```bash
gcloud config set project <PROJECT_ID>
gcloud config set compute/region <REGION>
gcloud config set compute/zone <ZONE>
```

다만 기본값을 설정해두는 것은 편리한 만큼 위험할 수도 있다.

잘못된 프로젝트나 리전에 명령을 실행할 수 있기 때문이다.  
따라서 중요한 명령을 실행하기 전에는 현재 설정을 확인하는 습관이 필요하다.

```bash
gcloud config list
kubectl config current-context
kubectl get namespace
```

## GitHub CLI로 저장소 생성하기

GitHub CLI를 사용하면 터미널에서 바로 GitHub 저장소를 생성하고 clone할 수 있다.

```bash
gh repo create <REPOSITORY_NAME> --public --clone
```

이 명령은 크게 두 가지 일을 한다.

1. GitHub에 새 원격 저장소를 생성한다.
2. 생성한 저장소를 현재 디렉터리 아래로 clone한다.

명령의 구성은 다음과 같다.

| 부분 | 의미 |
|---|---|
| `gh` | GitHub CLI 도구 |
| `repo create` | 새 저장소를 생성하는 하위 명령 |
| `<REPOSITORY_NAME>` | 생성할 저장소 이름 |
| `--public` | 공개 저장소로 생성 |
| `--clone` | 생성 직후 로컬에 clone |

관련 옵션도 함께 알아두면 좋다.

| 옵션 | 의미 |
|---|---|
| `--source <PATH>` | 기존 로컬 디렉터리를 원격 저장소와 연결 |
| `--push` | 로컬 커밋을 생성 직후 push |
| `--description "..."` | 저장소 설명 추가 |
| `--private` | 비공개 저장소 생성 |
| `--public` | 공개 저장소 생성 |
| `--internal` | 조직 내부 저장소 생성 |

GitHub 저장소를 처음 만들 때는 공개 범위를 신중하게 선택해야 한다.

학습 노트나 공개 가능한 예제는 public으로 둘 수 있지만, 실제 회사 프로젝트나 민감 정보가 포함된 저장소는 private 또는 internal로 관리해야 한다.

## `git remote -v`

`git remote`는 현재 로컬 저장소에 연결된 원격 저장소 목록을 관리하는 명령이다.

여기에 `-v` 또는 `--verbose` 옵션을 붙이면 원격 저장소 이름뿐만 아니라 실제 URL과 용도까지 함께 볼 수 있다.

```bash
git remote -v
```

출력 예시는 다음과 같다.

```bash
origin  https://github.com/<USER>/<REPOSITORY_NAME>.git (fetch)
origin  https://github.com/<USER>/<REPOSITORY_NAME>.git (push)
```

각 항목의 의미는 다음과 같다.

| 항목 | 의미 |
|---|---|
| `origin` | 원격 저장소의 별칭 |
| `https://github.com/...` | 원격 저장소 URL |
| `(fetch)` | 데이터를 가져올 때 사용하는 URL |
| `(push)` | 데이터를 올릴 때 사용하는 URL |

`-v` 없이 실행하면 원격 저장소 이름만 출력된다.

```bash
git remote
```

출력:

```bash
origin
```

그래서 로컬 저장소가 실제로 어떤 원격 저장소와 연결되어 있는지 확인할 때는 `git remote -v`를 자주 사용한다.

관련 명령은 다음과 같다.

```bash
git remote add <NAME> <URL>
git remote remove <NAME>
git remote set-url origin <NEW_URL>
git remote show origin
```

`git remote -v`는 “지금 이 로컬 저장소가 어디로 push/pull 하는가”를 확인하는 가장 기본적인 명령이다.

## `git symbolic-ref --short HEAD`

다음 명령은 현재 체크아웃된 브랜치 이름을 출력한다.

```bash
git symbolic-ref --short HEAD
```

예를 들어 `main` 브랜치에 있다면 다음처럼 출력된다.

```bash
main
```

명령의 구성은 다음과 같다.

| 부분 | 의미 |
|---|---|
| `git symbolic-ref` | 심볼릭 참조를 읽거나 설정하는 명령 |
| `HEAD` | 현재 작업 위치를 가리키는 특수 참조 |
| `--short` | 전체 경로 대신 짧은 이름만 출력 |

`HEAD`는 “지금 내가 어디에서 작업 중인가”를 가리키는 포인터다.

보통 `HEAD`는 특정 브랜치를 간접적으로 가리킨다.  
이처럼 다른 참조를 가리키는 참조를 심볼릭 참조라고 한다.

```bash
git symbolic-ref HEAD
```

출력:

```bash
refs/heads/main
```

```bash
git symbolic-ref --short HEAD
```

출력:

```bash
main
```

스크립트에서는 현재 브랜치 이름을 변수로 받아올 때 사용할 수 있다.

```bash
BRANCH=$(git symbolic-ref --short HEAD)
git push origin "$BRANCH"
```

비슷한 명령과의 차이는 다음과 같다.

| 명령 | 특징 |
|---|---|
| `git symbolic-ref --short HEAD` | 브랜치에 있을 때만 동작. detached HEAD 상태에서는 에러 발생 |
| `git branch --show-current` | 브랜치가 아니면 빈 문자열 반환 |
| `git rev-parse --short HEAD` | 브랜치 이름이 아니라 짧은 커밋 해시 반환 |

특정 커밋이나 태그를 직접 체크아웃한 detached HEAD 상태에서는 다음 오류가 발생할 수 있다.

```bash
fatal: ref HEAD is not a symbolic ref
```

스크립트에서 더 안전하게 처리하고 싶다면 `git branch --show-current`를 사용하는 것도 방법이다.

## `CLAUDE.md` 행동 규칙

이번 장에서 가장 중요하게 느껴진 부분 중 하나는 `CLAUDE.md`였다.

`CLAUDE.md`는 Claude Code가 프로젝트에서 작업할 때 참고하는 행동 규칙 문서이다.  
AI에게 “이 프로젝트에서는 이렇게 행동해야 한다”는 기준을 알려주는 역할을 한다.

예를 들어 다음과 같은 규칙을 정의할 수 있다.

1. 명령 실행 전 현재 상태를 확인한다.
   - 예: `kubectl get`, `gcloud config list`
2. 파일 변경 전 기존 내용을 먼저 읽는다.
3. 에러 발생 시 원인을 분석하고 해결 방안을 제시한 뒤 진행한다.
4. Manifest 작성 시 namespace를 명시한다.
   - 예: `<NAMESPACE>`
5. 모든 `kubectl` 명령에 context를 명시한다.
   - 예: `--context <KUBE_CONTEXT>`
6. 리소스 생성 또는 삭제 전에는 영향 범위를 먼저 설명한다.
7. 이미지 태그는 `latest`를 쓰지 않고 명시적 버전을 사용한다.
   - 예: `v0.1.0`
8. 토큰, 키, 비밀번호는 코드나 manifest에 하드코딩하지 않는다.
   - 환경 변수, Secrets, Secret Manager 등을 사용한다.

특히 Kubernetes 작업에서는 context와 namespace를 명시하는 규칙이 중요하다.

현재 터미널의 기본 context에 의존하면 의도하지 않은 클러스터에 명령을 실행할 수 있다.  
따라서 AI 에이전트가 실행하는 명령에는 대상 환경을 명확히 드러내는 것이 안전하다.

### 공통으로 필요해 보이는 규칙

여러 프로젝트에 공통으로 적용할 수 있는 규칙은 다음과 같다.

- 명령 실행 전 현재 상태를 확인한다.
- 파일 변경 전 기존 내용을 먼저 읽는다.
- 에러 발생 시 원인을 분석한 뒤 진행한다.
- 리소스 생성/삭제 전 영향 범위를 설명한다.
- 이미지 태그에 `latest`를 사용하지 않는다.
- 토큰, 키, 비밀번호를 하드코딩하지 않는다.
- 운영 환경에 영향을 줄 수 있는 명령은 사용자 승인 없이 실행하지 않는다.
- 삭제 명령은 실행 전 대상 리소스를 다시 조회한다.
- 적용 전 dry-run 또는 diff를 먼저 검토한다.
- 작업 완료 후 변경 사항과 검증 결과를 요약한다.

### 프로젝트별로 달라질 수 있는 규칙

반대로 다음 항목은 프로젝트별로 달라질 수 있다.

- 기본 namespace
- Kubernetes context
- 배포 대상 클러스터
- 이미지 레지스트리
- 브랜치 전략
- 배포 가능 시간대
- 승인 절차
- 롤백 방식

### 생각한 점

AI 에이전트에게 일을 맡기려면 먼저 행동 규칙을 정해야 한다.

사람에게도 프로젝트마다 작업 규칙과 운영 절차가 필요한 것처럼, AI에게도 프로젝트별 규칙이 필요하다.  
`CLAUDE.md`는 그 규칙을 문서로 고정해두는 역할을 한다.

결국 중요한 것은 AI가 무엇을 할 수 있는지가 아니라, **AI가 어떤 규칙 안에서 행동하도록 만들 것인가**이다.

## Cloud Build와 이미지 빌드

책에서는 Cloud Build를 사용해 이미지를 빌드하고 Artifact Registry에 올리는 흐름을 다룬다.

이 방식의 장점은 로컬 개발 환경에 Docker가 없어도 클라우드 환경에서 이미지를 빌드할 수 있다는 점이다.

일반적으로 이미지를 빌드하려면 로컬에 Docker가 설치되어 있어야 한다.  
하지만 빌드 환경을 클라우드나 CI/CD 시스템으로 옮기면 개발자 PC 환경에 덜 의존할 수 있다.

이런 방식은 다음과 같은 장점이 있다.

- 표준화된 빌드 환경 제공
- 개발자 로컬 환경 차이 감소
- 빌드 로그와 결과 추적 가능
- 이미지 태그 정책 적용 용이
- 승인된 파이프라인을 통한 이미지 생성
- 보안 정책 적용 가능

조직 내부 환경에서 비슷한 흐름을 만들고 싶다면 다음 도구들을 검토해볼 수 있다.

- GitHub Actions
- GitLab Runner
- Kubernetes 기반 CI Runner
- Kaniko
- BuildKit
- Harbor
- Nexus Repository

### 생각한 점

이미지 빌드 역시 개발자 로컬 환경에서만 처리할 것이 아니라, 표준화된 빌드 시스템에서 수행하는 것이 더 안정적일 수 있다.

특히 AI가 Dockerfile이나 배포 설정을 생성하는 환경에서는, 최종 이미지를 어디서 어떻게 빌드하고 검증할 것인지도 중요한 설계 요소가 된다.

## `kubectl port-forward`와 백그라운드 실행

`kubectl port-forward`는 로컬 포트와 Kubernetes 클러스터 안의 리소스를 연결할 때 사용한다.

예를 들면 다음과 같다.

```bash
kubectl port-forward svc/<SERVICE_NAME> -n <NAMESPACE> 8080:80
```

이 명령은 로컬 8080 포트를 Kubernetes 서비스의 80 포트로 연결한다.

다만 `kubectl port-forward`는 실행하면 계속 떠 있는 명령이다.  
터널을 유지해야 하므로 터미널이 해당 프로세스에 묶이고, 프롬프트가 돌아오지 않는다.

이때 명령 끝에 `&`를 붙이면 백그라운드에서 실행할 수 있다.

```bash
kubectl port-forward svc/<SERVICE_NAME> -n <NAMESPACE> 8080:80 &
```

각 부분의 의미는 다음과 같다.

| 부분 | 의미 |
|---|---|
| `kubectl port-forward` | 로컬 포트를 클러스터 리소스로 터널링 |
| `svc/<SERVICE_NAME>` | 대상 Kubernetes Service |
| `-n <NAMESPACE>` | 대상 namespace |
| `8080:80` | 로컬 8080 포트를 서비스 80 포트로 연결 |
| `&` | 명령을 백그라운드에서 실행 |

`&`를 붙이면 port-forward 터널은 백그라운드에서 유지되고, 터미널 프롬프트는 즉시 돌아온다.

그래서 같은 터미널에서 다음과 같은 명령을 이어서 실행할 수 있다.

```bash
curl localhost:8080
```

## `jobs`와 `kill %1`

백그라운드에서 실행한 작업은 `jobs` 명령으로 확인할 수 있다.

```bash
jobs
```

출력 예시는 다음과 같다.

```bash
[1]  running    kubectl port-forward svc/<SERVICE_NAME> -n <NAMESPACE> 8080:80
[2]  running    some-other-command
```

여기서 `[1]`은 1번 job을 의미한다.

1번 job을 종료하려면 다음 명령을 사용할 수 있다.

```bash
kill %1
```

`kill %1`의 의미는 다음과 같다.

| 부분 | 의미 |
|---|---|
| `kill` | 프로세스에 종료 시그널을 보내는 명령 |
| `%1` | 현재 셸의 1번 job을 가리키는 표기 |

PID를 직접 지정하는 방식과 job 번호를 지정하는 방식은 다르다.

```bash
kill 12345
```

이 명령은 PID가 `12345`인 프로세스를 종료한다.

```bash
kill %1
```

이 명령은 현재 셸의 1번 job을 종료한다.

관련 표기는 다음과 같다.

| 표기 | 의미 |
|---|---|
| `%1` | 1번 job |
| `%%` 또는 `%+` | 가장 최근 job |
| `%-` | 그 이전 job |
| `%문자열` | 해당 문자열로 시작하는 명령의 job |

`kubectl port-forward`처럼 백그라운드에서 실행하는 명령은 종료 방법까지 함께 알아두는 것이 좋다.

## `JOURNEY.md` 작성

이번 장에서는 커밋 전에 진행 상황을 기록할 `JOURNEY.md`를 만든다.

`JOURNEY.md`는 프로젝트를 진행하면서 다음과 같은 내용을 기록하는 파일이다.

- 무엇을 선택했는가
- 무엇을 설치했는가
- 어떤 설정을 변경했는가
- 어떤 문제를 겪었는가
- 어떻게 해결했는가
- 다음에 무엇을 해야 하는가

`CLAUDE.md`가 AI에게 주는 프로젝트 규칙 문서라면, `JOURNEY.md`는 사람이 프로젝트를 진행하며 남기는 작업 일지라고 볼 수 있다.

결과물만 남기면 나중에 왜 그런 선택을 했는지 알기 어렵다.  
하지만 진행 과정과 의사결정을 기록해두면 프로젝트를 다시 이해하거나 인수인계할 때 도움이 된다.

### 기록하면 좋은 내용

`JOURNEY.md`에는 다음 내용을 기록하면 좋다.

```markdown
# JOURNEY

## YYYY-MM-DD

### 오늘 진행한 작업

- 

### 선택한 도구와 이유

- 

### 변경한 설정

- 

### 발생한 문제

- 

### 해결 방법

- 

### 남은 작업

- 
```

### 생각한 점

문서화는 작업이 끝난 뒤에 따로 하는 일이 되면 쉽게 밀린다.

그래서 `JOURNEY.md`처럼 작업 흐름 안에서 자연스럽게 기록할 수 있는 문서가 필요하다.  
특히 AI와 함께 작업할 때는 AI가 어떤 변경을 했고, 왜 그런 방향으로 진행했는지 남겨두는 것이 중요하다.

## `/update-docs` 커스텀 스킬

책에서는 “지금까지 작업한 내용을 문서로 정리”하는 작업을 자동화하기 위해 `/update-docs` 커스텀 스킬을 만든다.

반복적으로 긴 프롬프트를 입력하는 대신, Claude Code의 커스텀 슬래시 명령을 만들어두면 한 명령으로 문서 갱신 작업을 실행할 수 있다.

커스텀 명령은 프로젝트 루트의 다음 경로에 파일로 등록할 수 있다.

```text
.claude/commands/update-docs.md
```

파일을 저장하면 Claude Code가 해당 명령을 인식하고, 이후 `/update-docs` 형태로 사용할 수 있다.

### `/update-docs`가 수행할 수 있는 일

`/update-docs`는 다음 흐름으로 설계할 수 있다.

1. 이번 작업에서 변경된 내용을 파악한다.
   - Git 커밋 기록
   - 도입된 도구
   - 변경된 설정
   - 결정 사항
   - 규칙 또는 컨텍스트 변경

2. 저장소 문서를 현재 상태에 맞게 갱신한다.
   - 존재하는 문서만 처리한다.
   - 없는 문서는 건너뛴다.

3. 새로 추가된 문서와 기존 문서의 변경 사항을 모두 반영한다.

4. 문서 변경 사항을 Git에 커밋한다.
   - 커밋 메시지는 이번 작업 요약을 기준으로 작성한다.

### 갱신 대상 문서 후보

프로젝트가 커지면 다음과 같은 문서를 갱신 대상으로 둘 수 있다.

| 문서 | 역할 |
|---|---|
| `JOURNEY.md` | 진행 현황, 도구 선택, 문제 해결 과정 |
| `CLAUDE.md` | AI 에이전트의 행동 규칙과 프로젝트 컨텍스트 |
| `docs/architecture-decisions.md` | 아키텍처 결정 사항 기록 |
| `claude-context/` | 현재 아키텍처나 프로젝트 상태 스냅샷 |
| `command-guardrails/` | 위험 명령 실행 절차와 가드레일 |

중요한 점은 문서 목록이 고정되어 있지 않아야 한다는 것이다.

프로젝트가 진행되면서 새 문서가 추가되더라도 `/update-docs`가 자연스럽게 갱신 대상에 포함할 수 있어야 한다.

### 생각한 점

`/update-docs` 같은 커스텀 스킬은 문서화를 작업 완료 절차에 포함시키는 좋은 방법이다.

AI에게 매번 “지금까지 한 작업을 정리해줘”라고 길게 말하는 대신, 프로젝트 표준에 맞는 명령을 만들어둘 수 있다.

예를 들면 다음과 같은 명령을 만들 수 있다.

```text
/update-docs
/update-journey
/update-release-notes
/update-handover
```

이런 방식은 문서화를 별도 업무가 아니라 작업 흐름의 일부로 만드는 데 도움이 된다.

## 내가 이해한 방식

Chapter 2를 읽고 Claude Code 환경 구성을 다음과 같이 이해했다.

```text
도구 설치
→ 권한과 실행 방식 결정
→ 현재 상태를 확인할 수 있는 정보 표시
→ 세션 관리와 되돌리기 전략 준비
→ Git과 Kubernetes 작업 환경 구성
→ CLAUDE.md로 AI 행동 규칙 정의
→ JOURNEY.md로 진행 과정 기록
→ /update-docs로 문서화 자동화
```

즉, 이 장의 핵심은 단순히 Claude Code를 실행하는 방법이 아니다.

AI 에이전트가 실제 프로젝트에서 작업할 수 있도록, 다음을 준비하는 과정이다.

- 어떤 권한으로 실행할 것인가
- 어떤 명령을 바로 실행해도 되는가
- 어떤 작업은 되돌릴 수 있고, 어떤 작업은 되돌리기 어려운가
- 어떤 규칙을 지키게 할 것인가
- 작업 결과를 어떻게 검증할 것인가
- 진행 과정과 의사결정을 어디에 남길 것인가

AI가 강력한 작업자가 될수록, 프로젝트에는 더 명확한 작업 규칙과 문서화 구조가 필요하다.

## 정리

Chapter 2는 환경 구성이라는 제목을 가지고 있지만, 단순히 도구를 설치하는 장은 아니었다.

이 장은 AI와 함께 인프라 작업을 하기 위해 필요한 **작업 환경, 권한, 안전장치, 문서화 구조**를 준비하는 내용에 가깝다.

이번 장을 읽고 가장 크게 남은 키워드는 다음과 같다.

- 자동 승인 모드는 편리하지만 권한 경계가 필요하다.
- `/btw`는 컨텍스트를 오염시키지 않고 보조 질문을 처리하는 데 유용하다.
- statusline은 현재 작업 환경을 한눈에 보여주는 안전장치가 될 수 있다.
- 세션 이어가기는 작업 흐름을 유지하는 데 도움이 된다.
- 되감기는 로컬 파일 변경에는 유용하지만 실제 인프라 변경까지 되돌려주지는 않는다.
- `CLAUDE.md`는 AI 에이전트의 행동 규칙을 정의하는 문서다.
- `JOURNEY.md`는 프로젝트 진행 과정과 의사결정을 기록하는 일지다.
- `/update-docs`는 문서화를 작업 흐름 안에 넣는 방법이 될 수 있다.

AI가 파일을 수정하고 명령을 실행할 수 있게 되면, 개발자의 역할은 단순히 명령을 직접 실행하는 것에서 끝나지 않는다.

앞으로 더 중요해질 역량은 다음과 같다고 느꼈다.

> AI가 안전하게 일할 수 있는 규칙을 설계하고,  
> 변경 사항을 검증하며,  
> 그 과정을 기록 가능한 흐름으로 만드는 능력

결국 AI 기반 인프라 작업에서 중요한 것은 자동화 자체가 아니다.

중요한 것은 자동화가 안전하고 추적 가능하며, 필요할 때 되돌릴 수 있는 구조 안에서 동작하도록 만드는 것이다.
